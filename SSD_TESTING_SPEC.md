# Спецификация: Полноценное тестирование SSD-накопителей (v2.0)

## Контекст

Этот документ — техническая спецификация для реализации модуля тестирования SSD в составе программы диагностики дисков. Модуль тестирования HDD уже реализован и работает корректно. Данная спецификация описывает **всё, что нужно знать** для реализации SSD-тестирования.

> **КРИТИЧЕСКИ ВАЖНО**: SSD нельзя тестировать как HDD. У них принципиально другая физика, другие метрики здоровья и другие паттерны деградации. Ниже — полная карта отличий и требований.

**Методологическая база:**
- **SNIA SSS PTS v2.0.2** — принципы предкондиционирования и steady-state (Purge → WIPC → WDPC → Test → Report)
- **JEDEC JESD218/219** — терминология TBW/DWPD, client workload модели
- **Практики iXBT (раздел id=11)** — прикладные кейсы, проверки "почему не как в паспорте", sanity-тесты
- **Документация fio / DiskSpd / nvme-cli / smartctl** — воспроизводимость, управление кэшированием

## 1. Фундаментальные отличия SSD от HDD (для корректной реализации)

### 1.1. Почему HDD-подход не работает для SSD

| Аспект | HDD | SSD |
|--------|-----|-----|
| «Поверхность» | Физические пластины — можно сканировать посекторно | Нет «поверхности» — ячейки NAND, понятие «скан поверхности» не имеет смысла в классическом виде |
| Бэд-блоки | Физический дефект магнитного слоя | Износ ячеек NAND (ограниченное число циклов P/E) |
| Скорость деградации | Обычно стабильна, резкое падение = механическая проблема | Постепенная, но с резкими ступенями (исчерпание SLC-кэша, wear leveling) |
| Время доступа | Зависит от позиции головки (мс) | Практически постоянное (мкс), но может расти при деградации контроллера |
| Запись | Безвредна для диагностики | **Каждая запись приближает смерть накопителя** — тесты записи расходуют ресурс! |
| Кэширование | Небольшой буфер, не влияет на тесты | SLC-кэш кардинально искажает результаты коротких тестов |
| TRIM | Не применимо | Критически важен — без TRIM деградация ускоряется, тесты дают искажённые результаты |
| Температура | Влияет мало | Троттлинг при перегреве — скорость может падать в 5-10 раз |
| **История записи** | Не влияет на результаты | **Критически влияет!** Новый (FOB) SSD показывает скорости на 20-50% выше, чем б/у. Без предкондиционирования результаты несравнимы между устройствами |

### 1.2. Ключевой принцип: не навреди

SSD имеет конечный ресурс записи (TBW — Total Bytes Written). Каждый тест с записью расходует этот ресурс. Поэтому:

- **Тесты только на чтение** — безопасны, можно запускать сколько угодно
- **Тесты с записью** — предупреждать пользователя о расходе ресурса
- **Деструктивные тесты** (полная перезапись, Secure Erase, Sanitize) — только с двойным подтверждением и предупреждением, запрет на системном диске

## 2. Архитектура модуля SSD-тестирования

### 2.1. Категории тестов (в порядке выполнения)

```
┌─────────────────────────────────────────────────────────────┐
│              SSD Testing Pipeline v2.0                       │
├─────────────────────────────────────────────────────────────┤
│  Phase 0: ДЕТЕКЦИЯ И ИДЕНТИФИКАЦИЯ                          │
│  ├── Определение типа (SATA / NVMe / USB)                   │
│  ├── Получение паспорта накопителя                           │
│  ├── Определение возможностей (TRIM, NCQ и т.д.)            │
│  └── Snapshot окружения (OS, CPU, RAM, драйвер, режим пит.) │
│                                                              │
│  Phase 1: ЗДОРОВЬЕ (неразрушающее, read-only)               │
│  ├── S.M.A.R.T. анализ                                      │
│  ├── Температурный мониторинг                                │
│  ├── Оценка оставшегося ресурса                              │
│  ├── Проверка прошивки (firmware)                             │
│  └── Фиксация host writes ДО тестов (для дельты)            │
│                                                              │
│  Phase 1.5: ПРЕДКОНДИЦИОНИРОВАНИЕ (опционально, SNIA PTS)   │
│  ├── Purge: Secure Erase / Sanitize / Format / blkdiscard   │
│  ├── WIPC: 2× ёмкости SEQ 128K записью (steady-state)       │
│  └── WDPC: прогон целевой нагрузки до стабилизации          │
│  (!) Без этой фазы результаты маркируются                    │
│      steady_state_valid=false                                │
│                                                              │
│  Phase 2: ПРОИЗВОДИТЕЛЬНОСТЬ (чтение — безопасно)            │
│  ├── Последовательное чтение                                 │
│  ├── Случайное чтение (4K, 8K, 16K, 128K)                   │
│  ├── Многопоточное чтение (QD1, QD4, QD16, QD32)            │
│  ├── Латентность чтения                                      │
│  └── Устойчивость скорости чтения на полном объёме           │
│                                                              │
│  Phase 3: ПРОИЗВОДИТЕЛЬНОСТЬ (запись — с предупр.)            │
│  ├── Последовательная запись                                  │
│  ├── Случайная запись (4K, 8K)                               │
│  ├── Смешанная нагрузка (70/30, 50/50 read/write)           │
│  ├── Тест SLC-кэша (запись до исчерпания кэша)              │
│  └── Sustained write (длительная непрерывная запись)          │
│                                                              │
│  Phase 4: СТАБИЛЬНОСТЬ (опционально, длительные)             │
│  ├── Стресс-тест (нагрузка + температура)                    │
│  ├── Тест целостности данных (write-read-verify)             │
│  └── Тест на стабильность латентности                        │
│                                                              │
│  Phase 5: ИТОГОВЫЙ ОТЧЁТ                                     │
│  ├── Фиксация host writes ПОСЛЕ тестов (дельта = расход)     │
│  ├── Общая оценка здоровья (0-100%)                          │
│  ├── Прогноз оставшегося времени жизни                       │
│  ├── Сравнение с эталонными показателями                     │
│  ├── Флаг steady_state_valid (было ли предкондиц.)           │
│  └── Рекомендации                                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2. Уровни тестирования (пользователь выбирает)

| Уровень | Название | Время | Запись | Что включено |
|---------|----------|-------|--------|-------------|
| 1 | **Быстрая проверка** | 1-3 мин | Нет | Phase 0 + Phase 1 + базовый Phase 2 |
| 2 | **Стандартное тестирование** | 10-20 мин | Минимальная | Phase 0-2 полностью + базовый Phase 3 |
| 3 | **Полное тестирование** | 30-60 мин | Умеренная | Phase 0-3 полностью |
| 4 | **Стресс-тестирование** | 1-4 часа | Значительная | Phase 0-4 полностью |
| 5 | **SNIA-подобное (лабораторное)** | 2-8 часов | Максимальная | Phase 0-4 + Phase 1.5 (Preconditioning). Для приёмки партий и сравнений |

## 3. Phase 0: Детекция и идентификация

### 3.1. Определение типа накопителя

```
Алгоритм определения:
1. Проверить WMI: Win32_DiskDrive → MediaType, InterfaceType
2. Для NVMe: проверить наличие в "NVMe" namespace
3. Для SATA SSD: RotationRate == 0 или == 1 (SSD не имеют RPM)
   - Дополнительно: проверить NominalMediaRotationRate через IDENTIFY DEVICE
4. Для USB SSD: InterfaceType == "USB", далее пробовать SAT passthrough
5. Определить протокол:
   - NVMe (PCIe) — самый быстрый, свои команды
   - AHCI/SATA — классический SATA SSD
   - USB (UASP/BOT) — внешний SSD, ограничения интерфейса
```

### 3.2. Получение паспорта накопителя

Собрать и отобразить:

```
- Производитель (Vendor)
- Модель (Model)
- Серийный номер (Serial)
- Версия прошивки (Firmware Revision)
- Ёмкость (в байтах и человекочитаемо)
- Интерфейс (SATA III / PCIe Gen3x4 / PCIe Gen4x4 / USB 3.x)
- Протокол (AHCI / NVMe 1.3 / NVMe 1.4 / NVMe 2.0)
- Поддержка TRIM: да/нет
- Поддержка NCQ: да/нет (и глубина очереди)
- Размер сектора: логический / физический (512 / 512e / 4Kn)
- Тип NAND: SLC / MLC / TLC / QLC / PLC (если доступно)
- Количество свободного места / общее
- Температура текущая
```

### 3.3. Snapshot окружения (Environment Snapshot)

Фиксируется перед каждым прогоном. Нужен для воспроизводимости и для флага `comparable` при сравнении прогонов.

```
Собрать и сохранить в отчёт:
- OS: версия, build, разрядность
- Kernel / Driver: версия драйвера хранения (StorPort, NVMe driver и т.п.)
- CPU: модель, количество ядер, частота
- RAM: объём, скорость
- Режим питания: "Высокая производительность" / "Сбалансированный" / другой
- Подключение SSD: слот (M.2, SATA-порт), через чипсет или CPU напрямую
- TRIM: включён/выключен (fsutil behavior query DisableDeleteNotify)
- Версии инструментов: если используются fio/DiskSpd/smartctl — записать версии
- Состояние шифрования: BitLocker активен/нет
- Антивирус: активен/отключён для тестовой папки
- Timestamp: начало/конец прогона, timezone
```

### 3.4. Windows-команды и API для детекции

```powershell
# WMI запрос
Get-WmiObject Win32_DiskDrive | Select Model, MediaType, InterfaceType, Size, SerialNumber, FirmwareRevision

# PowerShell — физические диски
Get-PhysicalDisk | Select FriendlyName, MediaType, BusType, HealthStatus, Size, FirmwareVersion

# NVMe-специфичное (Windows 10+)
Get-PhysicalDisk | Where-Object { $_.BusType -eq "NVMe" }

# Для SATA: DeviceIoControl с IOCTL_ATA_PASS_THROUGH
# Для NVMe: DeviceIoControl с IOCTL_STORAGE_QUERY_PROPERTY (StorageAdapterProtocolSpecificProperty)

# Проверка TRIM
fsutil behavior query DisableDeleteNotify
# 0 = TRIM включён, 1 = TRIM выключен

# Режим питания
powercfg /getactivescheme

# Версия драйвера хранения
Get-WmiObject Win32_PnPSignedDriver | Where-Object { $_.DeviceName -like "*NVMe*" -or $_.DeviceName -like "*AHCI*" }
```

### 3.5. Программные API (C# / C++ / Python)

```
Windows API:
- CreateFile("\\\\.\\PhysicalDriveN", ...) — открытие устройства
- DeviceIoControl() с кодами:
  - IOCTL_STORAGE_QUERY_PROPERTY — свойства накопителя
  - IOCTL_DISK_GET_DRIVE_GEOMETRY_EX — геометрия
  - IOCTL_ATA_PASS_THROUGH — ATA-команды (для SATA SSD)
  - IOCTL_STORAGE_PROTOCOL_COMMAND — NVMe-команды
  - IOCTL_SCSI_PASS_THROUGH — SCSI passthrough (USB SSD)

Библиотеки:
- C#: System.Management (WMI), P/Invoke для DeviceIoControl
- Python: wmi, pySMART, smartmontools (через subprocess)
- Утилиты: smartctl (smartmontools), nvme-cli
```

### 4.2. Расчёт оставшегося ресурса

```
Алгоритм:

1. Получить TBW из спецификации накопителя (по модели)
   - Если нет в базе — использовать эвристику:
     - Consumer TLC: ~600 TBW на 1 ТБ ёмкости
     - Consumer QLC: ~360 TBW на 1 ТБ ёмкости
     - Enterprise/Server: ~3000+ TBW на 1 ТБ ёмкости
   - Примечание: производители рассчитывают TBW по JEDEC JESD219 
     (client workload). Реальный ресурс зависит от паттерна нагрузки.

2. Получить Total Host Writes (из S.M.A.R.T. 241/F1 или NVMe Data Units Written)

3. Рассчитать:
   - consumed_tbw = total_host_writes (в ТБ)
   - remaining_tbw = rated_tbw - consumed_tbw
   - percentage_used = (consumed_tbw / rated_tbw) * 100
   
4. Рассчитать прогноз:
   - daily_write_rate = consumed_tbw / power_on_days
   - estimated_remaining_days = remaining_tbw / daily_write_rate
   
5. Скорректировать по S.M.A.R.T.:
   - Если Percentage Used (NVMe) или SSD Life Left (SATA) доступен — 
     использовать его как приоритетный индикатор
   - Если Reallocated Sectors > 0 — понизить прогноз на 20%
   - Если Uncorrectable Errors > 0 — понизить прогноз на 40%
   
6. Write Amplification Factor (WAF):
   - WAF = NAND_Writes / Host_Writes
   - Нормально: 1.0-3.0
   - Плохо: >5.0 (слишком много внутренней записи / garbage collection)
```

### 4.3. Общая оценка здоровья (Health Score)

```
Формула Health Score (0-100):

base_score = 100

# Критические факторы (каждый может снять до 40 баллов)
if reallocated_sectors > 0:
    base_score -= min(40, reallocated_sectors * 2)
    
if uncorrectable_errors > 0:
    base_score -= min(40, uncorrectable_errors * 5)
    
if program_fail_count > 0:
    base_score -= min(30, program_fail_count * 3)

if erase_fail_count > 0:
    base_score -= min(30, erase_fail_count * 3)

# Износ (до 30 баллов)
if percentage_used is not None:
    if percentage_used > 100:
        base_score -= 30
    elif percentage_used > 90:
        base_score -= 25
    elif percentage_used > 80:
        base_score -= 15
    elif percentage_used > 50:
        base_score -= 5

# Температура (до 15 баллов)
if temperature > 80:
    base_score -= 15
elif temperature > 70:
    base_score -= 10
elif temperature > 60:
    base_score -= 5

# Pending sectors (до 20 баллов)
if pending_sectors > 0:
    base_score -= min(20, pending_sectors * 4)

# Аварийные отключения (до 10 баллов)
if unsafe_shutdowns > 1000:
    base_score -= 10
elif unsafe_shutdowns > 100:
    base_score -= 5

# CRC ошибки (проблемы интерфейса, до 10 баллов)
if crc_errors > 0:
    base_score -= min(10, crc_errors)

# NVMe Critical Warning (до 30 баллов)
if critical_warning != 0:
    base_score -= 30  # Любой бит = серьёзная проблема

health_score = max(0, base_score)

# Категории:
# 90-100: Отлично (зелёный)
# 70-89:  Хорошо (жёлто-зелёный)
# 50-69:  Удовлетворительно (жёлтый)
# 30-49:  Плохо (оранжевый) — рекомендуется замена
# 0-29:   Критично (красный) — немедленная замена
```

### 4.4. Фиксация host writes ДО тестов

```
Перед началом любых тестов с записью — зафиксировать:
- NVMe: data_units_written (из Health Information Log)
- SATA: атрибут 241 (Total Host Writes) или 233 (Total LBAs Written)

После завершения тестов — зафиксировать снова.

delta_host_writes = after - before

Отобразить в отчёте:
- "Тестирование записало на SSD: X ГБ (Y% от оставшегося TBW)"
```

## 5. Phase 1.5: Предкондиционирование (SNIA PTS)

> Эта фаза **опциональна** для обычной диагностики, но **обязательна** для корректного сравнения SSD между собой (приёмка партий, выбор модели, бенчмарки "до/после").

### 5.1. Зачем это нужно

SSD показывает разную производительность в зависимости от "истории записи":
- **FOB (Fresh Out of Box)** — все ячейки чистые, SLC-кэш максимален, скорость максимальна
- **После эксплуатации** — часть ячеек грязная, garbage collection работает в фоне, скорость ниже
- **Steady State** — стабильное состояние после длительной нагрузки, скорость стабилизировалась

Без предкондиционирования **нельзя корректно сравнить** новый SSD Samsung с б/у SSD Kingston — результаты несопоставимы.

### 5.2. Workflow по SNIA PTS

```
Шаг 1: PURGE — возврат к состоянию, близкому к FOB
  ├── NVMe: nvme sanitize -a 2 /dev/nvme0n1 (или nvme format)
  ├── SATA: hdparm --security-erase /dev/sdX
  ├── Быстрый вариант: blkdiscard /dev/... (TRIM всего диска)
  ├── Windows: нет стандартной команды, использовать утилиту производителя
  │   или записать нулями весь диск (diskpart clean all)
  └── ⚠️ УНИЧТОЖАЕТ ВСЕ ДАННЫЕ! Двойное подтверждение обязательно.

Шаг 2: WIPC (Workload Independent Preconditioning)
  ├── Терминология SNIA: записать 2× user capacity
  │   последовательными блоками 128K
  ├── Пример для 1 ТБ SSD: записать 2 ТБ данных
  ├── Использовать случайные данные (не нули!),
  │   т.к. контроллеры могут сжимать нули
  └── Direct I/O обязательно

Шаг 3: WDPC (Workload Dependent Preconditioning) 
  ├── Прогон целевой нагрузки (того теста, что будет измеряться)
  ├── До достижения steady-state: 
  │   скорость/IOPS стабилизировалась (отклонение < 10% между раундами)
  └── Обычно 5-20 раундов по 1-5 минут

Шаг 4: MEASUREMENT WINDOW — только теперь запускаются замеры!
```

### 5.3. Флаги в отчёте

```
Для каждого тестового прогона фиксировать:

preconditioning:
  purge_performed: true/false
  purge_method: "nvme_sanitize" | "security_erase" | "blkdiscard" | "none"
  wipc_performed: true/false
  wipc_volume_written_gb: 2048
  wdpc_performed: true/false
  wdpc_rounds: 12
  steady_state_achieved: true/false
  steady_state_valid: true/false   ← КЛЮЧЕВОЙ ФЛАГ

Если steady_state_valid = false, отчёт содержит предупреждение:
"Результаты получены БЕЗ предкондиционирования по SNIA PTS.
 Сравнение с другими прогонами допустимо только как best-effort."
```

## 6. Phase 2: Тесты производительности (чтение)

### 6.1. Общие правила проведения тестов

```
ОБЯЗАТЕЛЬНЫЕ УСЛОВИЯ ПЕРЕД ТЕСТИРОВАНИЕМ:

1. Закрыть все приложения, активно использующие диск
2. Отключить антивирус на время теста (или исключить тестовый файл)
3. Подождать завершения фоновых операций (TRIM, garbage collection, 
   Windows indexing, Windows Defender scan)
4. Записать начальную температуру
5. Если SSD — системный диск, предупредить что результаты будут ниже 
   из-за фоновой нагрузки ОС
6. Для NVMe: проверить, подключён ли диск через PCIe напрямую 
   или через чипсет (влияет на латентность)

ПАРАМЕТРЫ ТЕСТИРОВАНИЯ:

- Выравнивание (alignment): 4 КБ (проверить, что раздел выровнен)
  Проверка: start offset раздела кратен physical sector size
  При misalignment — выдать warning (производительность деградирует!)
- Буферизация: Direct I/O (FILE_FLAG_NO_BUFFERING + FILE_FLAG_WRITE_THROUGH)
  Для DiskSpd: -Sh (отключить и software cache, и hardware write cache)
  Для fio: direct=1
- Тестовый файл: создавать на тестируемом разделе
- Размер тестового файла: минимум 1 ГБ, рекомендуется 16 ГБ 
  (чтобы выйти за пределы SLC-кэша большинства SSD)
- Количество проходов: минимум 3, брать медиану
- Прогрев (warm-up / ramp): 1 проход перед замерами (результат отбрасывается)
  Для fio: ramp_time=10s
  Для DiskSpd: -W10 (warmup 10 sec)
- Паттерн данных для записи: случайные данные (не нули!),
  т.к. контроллеры могут сжимать нули и показывать завышенные скорости
- Seed: фиксированный, записывается в отчёт для воспроизводимости
  Для fio: randseed=42, randrepeat=1
  Для DiskSpd: -z42
```

### 6.2. Пороговые критерии (link capacity based)

```
Автоматический расчёт ожидаемой максимальной скорости по интерфейсу:

SATA III:         600 MB/s теоретический → ~560 MB/s реальный максимум
PCIe Gen3 x1:   ~985 MB/s
PCIe Gen3 x2:  ~1970 MB/s
PCIe Gen3 x4:  ~3940 MB/s
PCIe Gen4 x4:  ~7880 MB/s
PCIe Gen5 x4: ~15760 MB/s
USB 3.0:         ~450 MB/s
USB 3.1 Gen2:  ~1050 MB/s
USB 3.2 Gen2x2: ~2100 MB/s

Пороги (настраиваемые):
- Seq Read  ≥ 0.80 × link_capacity (или ≥ 0.80 × паспортная, если задана): PASS
- Seq Read  ≥ 0.60 × link_capacity: WARN
- Seq Read  <  0.60 × link_capacity: FAIL (что-то не так)

Если пользователь ввёл паспортные скорости — сравнивать с ними:
- ≥ 0.85 × паспортная при совпадении условий теста: PASS
- < 0.85 × паспортная: WARN + проверить условия 
  (пометить, что паспортные условия обычно = FOB + CrystalDiskMark + 
   маленький тестовый файл)
```

### 6.3. Последовательное чтение (Sequential Read)

```
Параметры:
- Размер блока: 1 МБ (основной), дополнительно 128 КБ и 4 МБ
  (128 КБ = то, что использует CrystalDiskMark для Seq Q32T1)
- Queue Depth: 32 (для NVMe), 1-4 (для SATA)
- Потоков: 1 (основной), дополнительно 2 и 4
- Объём данных: весь тестовый файл (мин. 1 ГБ)
- Количество проходов: 3

Генерация команд:
  DiskSpd: diskspd -b1M -d60 -o32 -t1 -Sh -w0 X:\testfile.dat
  fio:     rw=read; bs=1m; iodepth=32; numjobs=1; direct=1; runtime=60

Измерять:
- Скорость (МБ/с): среднюю, минимальную, максимальную
- IOPS
- Латентность: среднюю, P95, P99, максимальную

Эталоны для сравнения:
- SATA SSD: 500-560 МБ/с (теоретический максимум SATA III)
- NVMe Gen3 x4: 2000-3500 МБ/с
- NVMe Gen4 x4: 5000-7400 МБ/с
- NVMe Gen5 x4: 10000-14000 МБ/с
```

### 6.4. Случайное чтение (Random Read)

```
Параметры:
- Размер блока: 4 КБ (основной!), дополнительно 8 КБ, 16 КБ, 128 КБ
- Queue Depth: тест при QD1, QD4, QD16, QD32
- Потоков: 1 (основной), дополнительно соответственно QD
- Область: весь тестовый файл (случайные LBA в пределах файла)
- Длительность: 30 секунд на каждую комбинацию

Генерация команд:
  DiskSpd: diskspd -b4K -d30 -o1 -t1 -Sh -r -w0 -z42 X:\testfile.dat
  fio:     rw=randread; bs=4k; iodepth=1; numjobs=1; direct=1; 
           runtime=30; randseed=42; randrepeat=1

Измерять:
- IOPS (главная метрика для random!)
- Латентность: среднюю, P95, P99, P99.9, максимальную
- МБ/с (вторичная метрика)

Пороговые критерии (4K QD1 Read Latency):
  NVMe: p50 ≤ 0.1 мс (pass), ≤ 0.3 мс (warn), > 0.6 мс (fail)
  SATA: p50 ≤ 0.2 мс (pass), ≤ 0.6 мс (warn), > 1.2 мс (fail)

Эталоны (Random 4K Read):
- SATA SSD: 80,000-100,000 IOPS (QD32)
- NVMe Gen3: 300,000-600,000 IOPS (QD32)
- NVMe Gen4: 500,000-1,000,000 IOPS (QD32)
- NVMe Gen5: 1,000,000-2,000,000 IOPS (QD32)

ВАЖНО: QD1 — самый важный для реальной пользовательской нагрузки!
Типичные значения QD1 Random 4K Read:
- SATA SSD: 8,000-12,000 IOPS
- NVMe Gen3: 12,000-20,000 IOPS
- NVMe Gen4: 15,000-25,000 IOPS
```

### 6.5. Тест устойчивости скорости чтения (Full Drive Read Sweep)

```
Назначение: проверить, что скорость стабильна на всём объёме диска
(у SSD должна быть стабильна, в отличие от HDD)

Параметры:
- Последовательное чтение блоками по 1 МБ
- Замер скорости каждые N мегабайт (N = total_size / 200, 
  чтобы получить 200 точек для графика)
- Записывать: позицию, скорость, латентность

Анализ результата:
- Стабильная линия = норма
- Провалы = возможные проблемы:
  - Регулярные провалы = тепловой троттлинг (сверить с температурой)
  - Единичные провалы = фоновые операции контроллера (GC, wear leveling)
  - Постоянное снижение = деградация (для SSD нетипично!)

Результат: график "скорость от позиции" (как в HD Tune)
```

### 6.6. Тест латентности чтения

```
Параметры:
- Случайное чтение 4 КБ, QD1
- 10,000 операций
- Записывать время каждой операции

Генерация команд:
  fio: rw=randread; bs=4k; iodepth=1; direct=1; 
       --output-format=json+  ← включает latency bins (QoS histogram)

Анализ:
- Построить гистограмму распределения латентности
- Вычислить: avg, median, P95, P99, P99.9, P99.99, max
- Отметить выбросы (outliers) > 10x средней

Нормы:
- SATA SSD: средняя ~0.05-0.1 мс, P99 < 0.5 мс
- NVMe SSD: средняя ~0.02-0.05 мс, P99 < 0.2 мс
- Выбросы > 10 мс — подозрительно (GC, троттлинг, проблемы)

Пороги QoS (по SNIA-подходу):
- NVMe: p99 > 2 мс = warn, p99 > 5 мс = fail
- SATA: p99 > 5 мс = warn, p99 > 10 мс = fail
```

## 7. Phase 3: Тесты производительности (запись)

### 7.1. Предупреждение пользователю

```
⚠️ ВАЖНО: Тесты записи расходуют ресурс SSD.
  
Перед запуском показать:
1. Текущий износ SSD (% от TBW)
2. Объём данных, который будет записан этим тестом
3. Сколько это в % от оставшегося TBW
4. Предупреждение: "Тесты записи сокращают срок службы SSD.
   Для нового SSD влияние минимально. 
   Для изношенного SSD (>80% TBW) — рекомендуется пропустить."
5. Запросить подтверждение

Для деструктивных операций (Purge, Secure Erase) — двойное подтверждение:
- Показать серийный номер и модель устройства
- Запросить ввод подтверждающей строки (название устройства или токен)
- Запрет на системном диске по умолчанию
```

### 7.2. Последовательная запись (Sequential Write)

```
Параметры аналогичны Sequential Read.

ВАЖНО: обязательно использовать FILE_FLAG_NO_BUFFERING + FILE_FLAG_WRITE_THROUGH,
иначе Windows кэширует запись и показывает скорость RAM, а не SSD!

ВАЖНО: использовать случайные данные (data_pattern=random), не нули.
Контроллеры SSD могут сжимать нули и показывать завышенные скорости.

Генерация команд:
  DiskSpd: diskspd -b1M -d60 -o32 -t1 -Sh -w100 X:\testfile.dat
  fio:     rw=write; bs=1m; iodepth=32; direct=1; runtime=60

Эталоны:
- SATA SSD: 400-530 МБ/с
- NVMe Gen3: 1500-3000 МБ/с
- NVMe Gen4: 3000-7000 МБ/с
```

### 7.3. Случайная запись (Random Write 4K)

```
Параметры аналогичны Random Read.

Генерация команд:
  DiskSpd: diskspd -b4K -d120 -o1 -t1 -Sh -r -w100 -z42 X:\testfile.dat
  fio:     rw=randwrite; bs=4k; iodepth=1; direct=1; runtime=120; randseed=42

Эталоны (Random 4K Write QD32):
- SATA SSD: 60,000-90,000 IOPS
- NVMe Gen3: 200,000-500,000 IOPS
- NVMe Gen4: 400,000-1,000,000 IOPS
```

### 7.4. Смешанная нагрузка (Mixed I/O)

```
Параметры:
- Случайные 4K операции
- Соотношение Read/Write: 70/30 (по умолчанию), также 50/50 и 65/35
  (65/35 часто упоминается в enterprise-контексте JEDEC)
- Queue Depth: 1, 4, 16, 32
- Длительность: 60-120 секунд

Генерация команд:
  DiskSpd: diskspd -b4K -d120 -o1 -t1 -Sh -r -w30 -z42 X:\testfile.dat
  fio:     rw=randrw; rwmixread=70; bs=4k; iodepth=1; direct=1; 
           runtime=120; randseed=42; --output-format=json+

Зачем: реальная нагрузка почти никогда не бывает чисто read или write.
Это самый приближённый к реальности синтетический тест.

Пороги:
- QoS: доля операций с латентностью > 10 мс не должна превышать 0.1%
- Регресс: ухудшение p99 > 20% от baseline = warn
```

### 7.5. ⭐ Тест SLC-кэша (критически важный для SSD!)

```
Назначение: определить размер SLC-кэша и реальную скорость записи 
после его исчерпания. ЭТО САМЫЙ ВАЖНЫЙ ТЕСТ ДЛЯ SSD!

Почему: Производители указывают скорость записи в SLC-кэш, 
которая в 2-10 раз выше реальной скорости записи в TLC/QLC NAND.

Алгоритм:
1. Убедиться, что на SSD достаточно свободного места (>50% для динамического кэша)
2. Непрерывная последовательная запись блоками по 1 МБ
3. Замер скорости каждые 100 МБ (или каждую секунду)
4. Продолжать, пока:
   a) Скорость не упадёт и не стабилизируется на новом уровне, ИЛИ
   b) Записано 2x размер SLC-кэша (обычно 10-30% ёмкости), ИЛИ
   c) Записано максимум 100 ГБ (ограничение для экономии ресурса)
5. После падения — продолжить запись ещё 30 секунд для определения 
   стабильной "после-кэшевой" скорости

Генерация команд (sustained write с логированием):
  DiskSpd: diskspd -b1M -d1800 -o32 -t1 -Sh -w100 -D1000 X:\testfile.dat
           (-D1000 = интервалы по 1000 мс для тайм-серии)
  fio:     rw=write; bs=1m; iodepth=32; direct=1; time_based=1; 
           runtime=1800; write_bw_log=slc_test; log_avg_msec=1000

Результат:
- График "скорость записи от объёма записанных данных"
- Размер SLC-кэша (точка перегиба = cliff)
- Скорость записи в SLC-кэш (до перегиба)
- Скорость записи в NAND (после перегиба)
- Коэффициент: SLC_speed / NAND_speed

Пороги:
- Cliff detected: падение > 50% от peak на интервале ≥ 60s = событие в отчёт
- Sustained_bw_min: warn если < 25% от peak_bw после стабилизации

Типичные значения:
- Samsung 990 Pro 2TB: кэш ~120 ГБ, SLC: 6500 МБ/с → NAND: 1500 МБ/с
- Бюджетный QLC NVMe 1TB: кэш ~12 ГБ, SLC: 3000 МБ/с → NAND: 300-500 МБ/с
- SATA TLC 1TB: кэш ~6-80 ГБ, SLC: 530 МБ/с → NAND: 100-300 МБ/с

⚠️ ПРЕДУПРЕЖДЕНИЕ О РАСХОДЕ РЕСУРСА:
Этот тест записывает 10-100 ГБ данных. 
Для SSD с TBW 600 TB это 0.001-0.01% ресурса (ничтожно).
Для SSD с TBW 150 TB это 0.006-0.06% ресурса (минимально).
Рассчитать и показать пользователю перед запуском.
```

### 7.6. Sustained Write (длительная запись)

```
Назначение: проверить поведение при длительной записи 
(включая тепловой троттлинг и восстановление GC)

Параметры:
- Последовательная запись 1 МБ блоками
- Длительность: 5-10 минут непрерывно
- Мониторить: скорость, температуру, латентность

Анализ:
- Этап 1: SLC-кэш (высокая скорость)
- Этап 2: Прямая запись в NAND (пониженная скорость)
- Этап 3: Возможный троттлинг (ещё ниже, если температура > порога)
- Этап 4: Возможные паузы на GC (провалы до 0 на графике)

Результат: временной график "скорость + температура от времени"
```

## 8. Phase 4: Тесты стабильности

### 8.1. Тест целостности данных (Write-Read-Verify)

```
Назначение: убедиться, что SSD корректно хранит записанные данные.
Это единственный способ обнаружить "тихую" порчу данных (silent corruption).

Алгоритм:
1. Генерировать блок данных с известным паттерном:
   - Использовать CSPRNG (криптографически стойкий генератор)
   - Записать CRC32/SHA256 каждого блока в память
2. Записать блоки на SSD (последовательно или случайно)
3. Синхронизировать (flush кэши: FILE_FLAG_WRITE_THROUGH)
4. Дождаться 1-5 секунд (дать контроллеру завершить внутренние операции)
5. Прочитать блоки обратно
6. Сравнить контрольные суммы

Варианты:
a) Быстрый (1 ГБ, последовательно)
b) Полный (весь свободный объём, случайный порядок чтения)
c) С задержкой (записать → подождать 10 минут → прочитать)

Размер блока: 4 КБ и 1 МБ (разные блоки проверяют разные пути в контроллере)

Любое несовпадение = 🔴 КРИТИЧЕСКАЯ ПРОБЛЕМА. 
Немедленно прекратить тест и предупредить пользователя.
```

### 8.2. Стресс-тест

```
Назначение: проверить стабильность под максимальной нагрузкой

Параметры:
- Смешанная нагрузка: 70% random read 4K + 30% random write 4K
- Queue Depth: 32 (или максимальный для устройства)
- Потоков: по числу ядер CPU (но не менее 4)
- Длительность: 30-60 минут
- Мониторинг каждые 5 секунд: скорость, IOPS, латентность, температура

Критерии провала:
- Температура > 85°C в течение > 1 минуты
- Латентность P99 > 100 мс
- Скорость упала > 80% от начальной (не из-за троттлинга)
- Ошибки I/O (любые)
- Зависание (таймаут > 30 секунд)

Результат: временной график всех метрик + вердикт "стабилен/нестабилен"
```

### 8.3. Тест стабильности латентности

```
Назначение: обнаружить периодические "подвисания" SSD (QoS проблемы)

Алгоритм:
- Случайное чтение 4K, QD1, 1 поток
- Длительность: 10 минут
- Записывать время КАЖДОЙ операции (precision timer)

Анализ:
- Построить timeline-график латентности
- Найти "шпили" (spikes) > 10x средней
- Проверить периодичность шпилей (если есть — это GC/wear leveling)
- Вычислить QoS метрики: % операций с латентностью > порога

Нормы QoS:
- 99% операций < 0.5 мс (SATA) / < 0.2 мс (NVMe)
- 99.9% операций < 2 мс (SATA) / < 1 мс (NVMe)
- 99.99% операций < 10 мс (SATA) / < 5 мс (NVMe)
```

## 9. Реализация ввода-вывода

### 9.1. Два подхода: встроенный движок и адаптеры

```
Программа поддерживает ДВА режима выполнения I/O:

A) ВСТРОЕННЫЙ ДВИЖОК (Native I/O Engine)
   - Прямые вызовы Windows API (CreateFile + DeviceIoControl)
   - Максимальный контроль над параметрами
   - Не требует установки внешних инструментов
   - Используется по умолчанию

B) АДАПТЕРЫ ДЛЯ ВНЕШНИХ ИНСТРУМЕНТОВ
   - DiskSpd (Windows) — генерация и парсинг команд
   - fio (Linux/Windows) — генерация job-файлов и парсинг JSON+
   - Используются для:
     * Воспроизведения результатов на другой машине
     * Сравнения с CrystalDiskMark (который использует DiskSpd-подобный бэкенд)
     * SNIA-подобных тестов (fio лучше подходит для steady-state)
     * Кроссплатформенности
```

### 9.2. Windows API для прямого I/O (встроенный движок)

```c
// Открытие устройства для прямого I/O
HANDLE hFile = CreateFileW(
    L"\\\\?\\C:\\TestFile.dat",  // или "\\\\.\\PhysicalDrive0" для raw
    GENERIC_READ | GENERIC_WRITE,
    FILE_SHARE_READ | FILE_SHARE_WRITE,
    NULL,
    CREATE_ALWAYS,  // или OPEN_EXISTING для чтения
    FILE_FLAG_NO_BUFFERING |      // Обход файлового кэша Windows
    FILE_FLAG_WRITE_THROUGH |     // Запись напрямую на диск
    FILE_FLAG_OVERLAPPED,         // Для асинхронного I/O (queue depth > 1)
    NULL
);

// ВАЖНО для FILE_FLAG_NO_BUFFERING:
// - Размер буфера должен быть кратен размеру сектора (обычно 512 или 4096)
// - Смещение должно быть кратно размеру сектора
// - Буфер должен быть выровнен по границе сектора:
LPVOID buffer = VirtualAlloc(NULL, bufferSize, MEM_COMMIT, PAGE_READWRITE);

// Для точного измерения времени:
LARGE_INTEGER freq, start, end;
QueryPerformanceFrequency(&freq);
QueryPerformanceCounter(&start);
// ... I/O операция ...
QueryPerformanceCounter(&end);
double elapsed_us = (double)(end.QuadPart - start.QuadPart) / freq.QuadPart * 1000000.0;
```

### 9.3. Асинхронный I/O для Queue Depth > 1

```c
// Для эмуляции QD > 1 нужен overlapped I/O:
// Предпочтительный вариант — I/O Completion Port:
HANDLE hIocp = CreateIoCompletionPort(hFile, NULL, 0, 0);

// Запуск операции:
OVERLAPPED ov = {0};
ov.Offset = (DWORD)(offset & 0xFFFFFFFF);
ov.OffsetHigh = (DWORD)(offset >> 32);
ReadFile(hFile, buffer, blockSize, NULL, &ov);

// Ожидание завершения:
DWORD bytesTransferred;
ULONG_PTR key;
LPOVERLAPPED pOv;
GetQueuedCompletionStatus(hIocp, &bytesTransferred, &key, &pOv, INFINITE);
```

### 9.4. Адаптер DiskSpd (Windows)

```
Генерация команды из профиля:

profile = {
  rw: "randread", bs: "4K", iodepth: 32, numjobs: 4,
  runtime: 300, ramp: 20, direct: true, cache: "disable_all",
  seed: 42, data_pattern: "random", target: "X:\\testfile.dat"
}

→ Генерируемая команда:
diskspd -b4K -d300 -W20 -o32 -t4 -Sh -r -w0 -z42 -L X:\testfile.dat

Маппинг параметров:
  -b{bs}        : размер блока
  -d{runtime}   : длительность (секунды)
  -W{ramp}      : warmup (секунды)
  -o{iodepth}   : outstanding I/O (queue depth)
  -t{numjobs}   : потоки
  -Sh            : отключить все кэши (software + hardware write cache)
  -Su            : отключить только software cache
  -r             : random (если rw содержит "rand")
  -w{write_pct}  : процент записи (0=read, 100=write, 30=70/30 mix)
  -z{seed}       : random seed
  -L             : включить latency stats
  -D{interval}   : интервал (мс) для тайм-серии

Парсинг результатов: DiskSpd выводит текстовый отчёт → парсить секции
"total", "Read IO", "Write IO", latency percentiles.
```

### 9.5. Адаптер fio (Linux/Windows)

```
Генерация job-файла из профиля:

[global]
ioengine=windowsaio    ; (или libaio для Linux)
direct=1
bs=4k
iodepth=32
numjobs=4
runtime=300
ramp_time=20
time_based=1

rw=randrw

Запуск:
  fio jobfile.fio --output-format=json+ --output=result.json

Парсинг: JSON+ содержит полные latency bins (QoS histogram) — 
это главное преимущество fio для tail-latency анализа.
```

## 10. Температурный мониторинг (непрерывный, во время всех тестов)

```

- < 50°C: Нормально (зелёный)

- 70-80°C: Критическая (красный), SSD снижает скорость (throttling)
- > 80°C: ПРИОСТАНОВИТЬ ТЕСТ, дождаться охлаждения до 60°C

Визуализация:
- На всех графиках производительности — вторая ось с температурой
- Это позволяет наглядно видеть корреляцию троттлинга и падения скорости
```

## 12. Сравнение прогонов (Run A vs Run B)

```
Назначение: сравнить два прогона (до/после прошивки, разные SSD, разные настройки)

Алгоритм:
1. Загрузить report.json обоих прогонов
2. Проверить совместимость:
   - Одинаковый профиль тестирования? Если нет → comparable=false
   - Одинаковое окружение? Если нет → comparable=limited (+ список различий)
   - Одинаковый seed? Если нет → comparable=limited
   - Оба steady_state_valid? Если нет → comparable=best_effort
3. Вычислить дельты:
   - Δ throughput (МБ/с и %)
   - Δ IOPS (и %)
   - Δ p99 latency (мкс и %)
   - Δ host writes / percentage_used
4. Оценить значимость:
   - |Δ| < 3% → "в пределах погрешности"
   - |Δ| 3-10% → "незначительное изменение"
   - |Δ| 10-30% → "заметное изменение" (warn если ухудшение)
```

## 13. Обработка ошибок и Edge Cases

### 13.1. Специфичные ситуации

```
1. SSD в режиме Read-Only:
   - Некоторые SSD при исчерпании ресурса переходят в read-only
   - Проверять перед тестами записи: попробовать записать 1 блок
   - Если ошибка записи — проинформировать, запустить только read-тесты

2. SSD за USB-мостом:
   - Многие S.M.A.R.T. атрибуты недоступны
   - Скорость ограничена интерфейсом (USB 3.0 = ~450 МБ/с, USB 3.2 Gen2 = ~1050 МБ/с)
   - Предупредить: "Результаты ограничены пропускной способностью USB"
   - SAT passthrough может не работать — обработать gracefully

3. Системный SSD (загрузочный):
   - ОС постоянно читает/пишет → результаты менее точные
   - Предупредить пользователя
   - Рекомендовать загрузку с WinPE для чистого теста
   - Увеличить количество проходов для усреднения
   - Запрет Purge/Secure Erase на системном диске

4. Зашифрованный SSD (BitLocker, VeraCrypt):
   - Тесты через файловую систему включают overhead шифрования
   - Предупредить: "Шифрование активно, результаты включают overhead"
   - Зафиксировать в environment snapshot
   
5. NVMe через чипсет vs напрямую через CPU:
   - Через чипсет: +2-5 мкс латентности, чуть ниже пиковая скорость
   - Определить слот (если возможно) и отметить в отчёте

6. Полный SSD (мало свободного места):
   - SLC-кэш уменьшается (динамический кэш зависит от свободного места!)
   - Тестовый файл может не поместиться
   - Минимум свободного места для теста: 2 ГБ (быстрая проверка), 20 ГБ (полный тест)
   - Предупредить и адаптировать размер тестового файла

7. SSD с Host Memory Buffer (HMB):
   - Бюджетные NVMe без DRAM используют оперативную память хоста
   - Результаты зависят от загрузки RAM
   - Определить по отсутствию DRAM в спецификации (если доступно)

8. RAID-массив из SSD:
   - Не тестировать массив как один диск — результаты будут от RAID-контроллера
   - Предупредить: "Обнаружен RAID. Для корректного тестирования 
     отдельных SSD их нужно отключить от массива."

9. Misalignment:
   - Проверить, что start offset раздела кратен physical sector size
   - При несовпадении — warn + объяснение (сильно влияет на 4K random!)
```

### 13.2. Обработка ошибок I/O

```
Типы ошибок и реакция:

1. ERROR_CRC (23): Ошибка контрольной суммы
   → Ошибка интерфейса (кабель, порт). Повторить 3 раза. 
     Если повторяется — отметить в отчёте, рекомендовать проверить подключение.

2. ERROR_READ_FAULT (30) / ERROR_WRITE_FAULT (29):
   → Ошибка чтения/записи. Записать LBA. 
     Если единичная — продолжить. 
     Если массовая (>10) — прервать тест, отметить как критическую проблему.

3. ERROR_NOT_READY (21):
   → Устройство не готово. Подождать 5 сек, повторить.
     Если повторяется — SSD возможно в режиме восстановления после сбоя.

4. ERROR_IO_DEVICE (1117):
   → Общая ошибка устройства. Самая серьёзная. 
     Записать контекст. Если повторяется — прекратить тест.

5. Таймаут (WAIT_TIMEOUT):
   → Операция не завершилась за 30 секунд.
     Прервать операцию. Если единичная — предупредить.
     Если повторяется — контроллер SSD не отвечает (очень плохо).

Общий принцип: НИКОГДА не игнорировать ошибки I/O тихо!
Каждая ошибка — в лог с timestamp, LBA, типом операции.
```

## 14. Визуализация (рекомендуемые графики)

### 14.1. Обязательные графики

```
1. Health Dashboard (одна страница):
   - Круговая диаграмма/gauge: общий Health Score (0-100)
   - Прогресс-бар: износ NAND (% TBW)
   - Индикатор температуры
   - Ключевые S.M.A.R.T. метрики

2. Sequential Performance (bar chart):
   - Столбцы: Seq Read, Seq Write, Mixed R/W
   - С линией эталона (link capacity / спецификация производителя)

3. Random IOPS vs Queue Depth (line chart):
   - Ось X: QD (1, 4, 16, 32)
   - Ось Y: IOPS
   - Линии: Read, Write, Mixed R, Mixed W

4. SLC Cache Exhaustion (line chart) — ⭐ САМЫЙ ИНФОРМАТИВНЫЙ
   - Ось X: Записано данных (ГБ)
   - Ось Y (левая): Скорость записи (МБ/с)
   - Ось Y (правая): Температура (°C)
   - Аннотация: точка перегиба = размер SLC-кэша
   - Аннотация: cliff = момент падения скорости

5. Latency Distribution (histogram + CDF):
   - Гистограмма: сколько операций попало в каждый диапазон латентности
   - CDF (кумулятивная): % операций с латентностью < X

6. Full Drive Read Sweep (line chart):
   - Ось X: Позиция на диске (ГБ)
   - Ось Y: Скорость чтения (МБ/с)
   - Для SSD должна быть ~ровная линия

7. Sustained Write Performance (line chart):
   - Ось X: Время (минуты)
   - Ось Y (левая): Скорость записи (МБ/с)
   - Ось Y (правая): Температура (°C)
   - Показывает фазы: SLC-кэш → direct NAND → throttling

8. Comparison Chart (при наличии двух прогонов):
   - Bar chart с парными столбцами Run A / Run B
   - Цветовая индикация: зелёный = улучшение, красный = ухудшение
```

## 15. Рекомендуемые утилиты для валидации результатов

```
Для проверки корректности реализации — сравнить результаты с:

1. CrystalDiskMark 8.x (самый популярный бенчмарк)
   - Сравнить: Seq Read/Write, Random 4K QD1 и QD32
   - Настройки: 16 ГБ тестовая область, 5 проходов
   - Примечание: CDM Seq Q32T1 использует bs=128K, а не 1M!
   - Результаты с тегом benchmark_class="gui_synthetic", comparability="limited"

2. CrystalDiskInfo (S.M.A.R.T.)
   - Сравнить: все S.M.A.R.T. атрибуты и значения

3. AS SSD Benchmark (дополнительная проверка)
   - Сравнить: 4K-64Thrd, Seq, Access Time

4. HD Tune Pro (для full drive sweep)
   - Сравнить: скорость vs позиция

5. Утилиты производителя:
   - Samsung Magician (для Samsung)
   - WD Dashboard (для WD/SanDisk)
   - Kingston SSD Manager (для Kingston)
   - Intel SSD Toolbox / Solidigm Synergy (для Intel/Solidigm)
   
6. smartmontools (smartctl) — эталон для S.M.A.R.T.
   - smartctl -a /dev/sdX (или smartctl -a /dev/nvmeN)
   - Сохранять полный вывод smartctl -x как raw-артефакт
```
