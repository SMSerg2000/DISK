# Проблема: извлечение Power-On Hours из SandForce-контроллеров

## Диск
- **Модель:** KINGSTON SKC300S37A60G (60 GB SSD)
- **Серийный:** 50026B723B02D7BB
- **Прошивка:** 507KC4
- **Контроллер:** SandForce SF-2281
- **Стандарт:** ATA8-ACS | ACS-2 Revision 3

## Суть проблемы

SMART атрибут **ID 9 (Power-On Hours)** содержит packed raw-значение (6 байт),
из которого мы не можем корректно извлечь реальные часы наработки.

## Данные от CrystalDiskInfo (корректный результат)
```
ID 09  Часы работы    Current=19  Worst=19  Raw=288ADE0011565
Общее время работы: 71013 ч
```

**71,013 часов = 8.1 лет** — совпадает с реальным возрастом диска.

Заметим: **0x11565 = 71,013** — часы находятся в нижних 2.5 байтах raw-значения (hex: ...11565).

## Данные от нашей программы (разные сессии)

### Сессия 1 (2026-03-25 16:35)
```
ID 9   Raw = 229,385,613,415,776
Hex:    0x????????44E2
Low16:  0x44E2 = 17,634 → отображали "2 г 0 мес" (НЕВЕРНО)
```

### Сессия 2 (2026-03-25 17:57)
```
ID 9   Raw = 265,523,468,244,322
```

### Сессия 3 (2026-03-26 15:30)
```
ID 9   Raw = 266,554,260,395,383
Hex:    0xF27DA0011577
Low16:  0x1577 = 5,495 → отображали "7 мес" (НЕВЕРНО)
```

## Ключевое наблюдение

1. **Low16 (нижние 2 байта) МЕНЯЕТСЯ** между сессиями: 17,634 → 5,495.
   Это НЕ может быть кумулятивный счётчик часов.

2. **CrystalDiskInfo извлекает 71,013** из raw-значения, где `0x11565 = 71013`.
   Но 0x11565 — это 17 бит (> 2 байт), значит это не простое low16.

3. **Верхние байты raw тоже меняются** динамически — SandForce использует
   их для внутренних счётчиков (ECC коррекции, soft errors и т.д.)

## Как SandForce пакует данные

SandForce SF-2281 использует нестандартный формат raw-значений для многих атрибутов.
6 байт raw содержат НЕСКОЛЬКО подсчётчиков:

```
Формат (предположительный, нужна верификация):
Byte[5] Byte[4] Byte[3] Byte[2] Byte[1] Byte[0]
  |       |       |       |       |       |
  +--- вспомогат. ---+    +-- основное значение --+
  (динамически          (для ID 9 = часы наработки)
   меняющиеся)
```

Но **граница между основным и вспомогательным** значением не на 16 битах
(как мы предполагали), а возможно на 24 или другом количестве бит.

## Текущий workaround

Для атрибутов 5, 196, 197, 198 мы используем `raw & 0xFFFFFFFF` (нижние 4 байта)
для определения критичности — это работает (ложный CRITICAL пропал).

Для Power-On Hours пока **не отображаем** наработку если `raw > 1,000,000`
(признак packed data). Ждём информацию о корректном методе извлечения.
