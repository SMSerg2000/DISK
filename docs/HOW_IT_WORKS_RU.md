## 1. Философия и общая архитектура

**Главный принцип проекта — ноль внешних зависимостей для доступа к дискам.** Единственная сторонняя библиотека — PySide6 (GUI). Вся работа с накопителями ведётся напрямую через `ctypes` + `kernel32.dll`: `CreateFileW` для открытия устройства и `DeviceIoControl` для команд драйверу. Никаких pywin32, wmi-обёрток или внешних утилит (PowerShell/WMI используется только как самый последний fallback для NVMe).

### Слои

```
┌─────────────────────────────────────────────────────────┐
│  GUI (PySide6)  — main_window, smart_table, panels       │
│      │  QThread-воркеры (фоновые задачи)                 │
│      ▼                                                    │
│  CORE (чистый Python + ctypes)                           │
│   ├─ winapi          — DeviceHandle, AlignedBuffer, lock │
│   ├─ drive_enumerator— перечисление, эвристики           │
│   ├─ smart_ata       — ATA legacy + SAT + IDENTIFY       │
│   ├─ smart_usb_nvme  — JMicron/ASMedia/Realtek мосты     │
│   ├─ smart_nvme      — NVMe fallback-цепочка + WMI        │
│   ├─ health_assessor — Health Score, TBW, WAF            │
│   ├─ benchmark       — измерение производительности      │
│   └─ surface_scan    — сканирование поверхности          │
│      │                                                    │
│      ▼                                                    │
│  WINDOWS API (kernel32.dll) — CreateFileW/DeviceIoControl│
│      │                                                    │
│      ▼                                                    │
│  \\.\PhysicalDriveN  /  \\.\C:  /  \\?\Volume{GUID}       │
└─────────────────────────────────────────────────────────┘
```

**Контракт между слоями** — модуль `core/models.py`: только `dataclass`/`Enum` (`DriveInfo`, `SmartAttribute`, `NvmeHealthInfo`, `HealthStatus`, `BenchmarkResult`, `SurfaceScanResult`), **без единой зависимости от ctypes**. Это позволяет тестировать backend и менять GUI независимо.

**Поток данных при чтении SMART** (упрощённо):

```
Выбор диска в GUI → QThread-воркер → read SMART (ATA/USB/NVMe)
   → assess_*_health() (оценка) → сигнал в главный поток → отрисовка таблицы + бейджа здоровья
```

## 2. Низкоуровневый слой доступа к диску

Файл: `core/winapi.py`. Здесь живут две ключевые обёртки — `DeviceHandle` и `AlignedBuffer` — и логика блокировки томов.

### 2.1. `DeviceHandle` — контекстный менеджер вокруг хэндла диска

```python
with DeviceHandle(drive_number, read_only=False, flags=0) as h:
    data = h.ioctl(IOCTL_CODE, in_struct, out_size)
```

**Открытие** (`__enter__`):
- Путь: `\\.\PhysicalDriveN` (или произвольный `device_path`, например `\\.\C:` или `\\?\Volume{GUID}`).
- **Уровень доступа** — главное правило: `read_only=True → GENERIC_READ` (хватает для перечисления, дескриптора, чтения NVMe-телеметрии, surface scan на чтение); `read_only=False → GENERIC_READ | GENERIC_WRITE` (нужно для SMART ATA pass-through, SAT-команд, записи). Write требует прав администратора, поэтому по умолчанию берётся минимум.
- Share mode всегда `FILE_SHARE_READ | FILE_SHARE_WRITE` — иначе открытый диск заблокировал бы доступ системе.
- `OPEN_EXISTING` (устройство уже существует).

**Проверка `INVALID_HANDLE_VALUE` — тонкий момент.** В WinAPI это `(HANDLE)-1`, что на 64-битной системе равно `0xFFFFFFFFFFFFFFFF`, а не `-1`. Поэтому используется `_INVALID_HANDLE = ctypes.c_void_p(-1).value`, и проверка делается «с запасом» (на `None`, `0`, `-1` и беззнаковое представление).

**Типизированные исключения** по `GetLastError()`:
- код 5 (Access Denied) → `AdminPrivilegeRequired` («запустите от администратора»);
- код 2 (File Not Found) → `DriveNotFound` (нормальный сигнал «конец сканирования»);
- иначе → `DiskAccessError`.

Эта типизация критична для перечисления: разные исключения = разная реакция.

**Три метода IOCTL** (драйверы ждут данные по-разному):
- `ioctl(code, in_struct, out_size)` — вход = ctypes-структура (размер через `ctypes.sizeof`, **не хардкод**), отдельный выходной буфер. Возвращает ровно столько байт, сколько записал драйвер (`bytes_returned`).
- `ioctl_raw(code, in_bytes, out_size)` — вход = сырые байты (для SAT/SCSI CDB, собираемых `struct.pack_into`).
- `ioctl_inplace(code, buffer)` — **один буфер на вход и выход** (NVMe-драйверы этого требуют).

**Сырой I/O** для бенчмарка и surface scan: `read/write` (через `ReadFile`/`WriteFile` — сами двигают файловый указатель ОС), `read_at/write_at` (= `seek` + `read/write`), `seek` (через `SetFilePointerEx`, 64-битный offset — работает на дисках > 2 ТБ).

### 2.2. `AlignedBuffer` — выровненный буфер для direct I/O

**Проблема:** флаг `FILE_FLAG_NO_BUFFERING` обходит кэш Windows (нужно для честного бенчмарка и surface scan — читаем «начисто»), но требует, чтобы буфер был **выровнен по размеру сектора** (512 или 4096 байт). Обычный `(c_ubyte * N)()` такой гарантии не даёт.

**Решение:** память берётся через `VirtualAlloc` — она всегда выровнена по границе страницы (4096 байт), чего с запасом хватает. Освобождение — `VirtualFree(..., MEM_RELEASE)` в `free()`/`__exit__`. Именно поэтому методы `read/write` принимают сырой указатель `buf.ptr`, а не Python-bytes.

### 2.3. Блокировка томов — `lock_and_dismount_volumes` / `VolumeLockResult`

Перед **деструктивной** записью (surface scan / бенчмарк) тома на диске блокируются и размонтируются. Иначе Windows либо запретит запись в область смонтированного раздела, либо запись «мимо» файловой системы рассинхронизирует её кэш и повредит ФС.

Алгоритм ищет тома **двумя способами**: перебором букв A–Z и через `FindFirstVolumeW`/`FindNextVolumeW` (ловит скрытые тома без буквы — EFI, Recovery, MSR). Для каждого тома через `IOCTL_VOLUME_GET_VOLUME_DISK_EXTENTS` проверяется, что он на **нужном** физическом диске (номер диска по offset 8), затем `FSCTL_LOCK_VOLUME` + `FSCTL_DISMOUNT_VOLUME`.

**Принцип fail-closed** — ключевое правило безопасности. Функция возвращает `VolumeLockResult{handles, failed_volumes}`. Если `failed_volumes` непустой (хоть один том не удалось надёжно заблокировать) — **деструктивная операция обязана отказаться**. Лучше не начать, чем повредить активную ФС. Сама функция не бросает исключение — решение принимает вызывающий код. Хэндл перечисления томов закрывается в `finally` (`FindVolumeClose`) — нет утечки даже при исключении в цикле.

`is_system_drive(drive_number)` — отдельная проверка «это диск с томом C:?» (открывает `%SystemDrive%` с доступом 0 и сравнивает номер диска). Используется для дополнительного предупреждения перед записью на системный диск.

## 3. Перечисление дисков

Файл: `core/drive_enumerator.py`, функция `enumerate_drives()`.

Перебираются `PhysicalDrive0..31` (`MAX_PHYSICAL_DRIVES = 32`), **read-only**. Несуществующий диск → `DriveNotFound` → `continue` (дыры в нумерации обрабатываются корректно). Каждый диск изолирован в `try/except`: один проблемный диск не роняет всё перечисление.

### 3.1. Дескриптор

`IOCTL_STORAGE_QUERY_PROPERTY` → буфер `STORAGE_DEVICE_DESCRIPTOR`. Дескриптор содержит не сами строки, а **смещения** до них внутри буфера (VendorIdOffset, ProductIdOffset, SerialNumberOffset, ProductRevisionOffset) + BusType. Строки извлекаются от смещения до нулевого байта. Модель = `vendor + " " + product`. Если дескриптор не достаётся (кривой USB-мост) — подставляются fallback-значения, диск **не теряется**.

### 3.2. Определение ёмкости — три метода

По убыванию надёжности, первый ненулевой результат побеждает:
1. `IOCTL_DISK_GET_LENGTH_INFO` — чистый int64, самый прямой;
2. `IOCTL_DISK_GET_DRIVE_GEOMETRY_EX` — поле `DiskSize` (offset 24);
3. `IOCTL_STORAGE_READ_CAPACITY` — идёт через storage stack, обходит disk class driver.

**Зачем третий метод:** disk-level IOCTL иногда падают с **error 1117** (ERROR_IO_DEVICE) на некоторых USB-мостах и RAID-контроллерах. Storage-level запрос обходит проблемный слой. Если всё провалилось — ёмкость 0 (GUI покажет «0.0 GB» вместо краша).

### 3.3. Цепочка эвристик определения интерфейса (строго по порядку)

Это сердце модуля. Базовый `interface = _bus_type_to_interface(bus_type)`, затем:

**Шаг 1. Детект гипервизора** — `_detect_hypervisor(model)`. Распознаёт виртуальные диски по строке модели (от специфичного к общему): VirtIO/Red Hat → «KVM/QEMU (VirtIO)», QEMU, Msft/Microsoft Virtual → Hyper-V, VMware, VBOX → VirtualBox, Xen/Citrix, Parallels, Google PersistentDisk, Amazon Elastic → AWS EBS. При совпадении → `InterfaceType.VIRTUAL`, имя гипервизора в `DriveInfo.hypervisor`.
*Почему:* у виртуальных дисков нет физического SMART — это абстракция гипервизора над LVM/NFS/Ceph/RAID. Пометка позволяет показать осмысленное сообщение вместо «Unknown».

**Шаг 2. NVMe по модели** (только если `interface == UNKNOWN`) — `_looks_nvme_model(model)`. OEM-драйверы Intel RST/VMD возвращают нестандартный bus_type, и интерфейс приходится угадывать. Логика:
- **Сначала исключение:** модель начинается с `MZ7` или `MZN` → НЕ NVMe (это Samsung **SATA**: MZ7* = 2.5" 850/860/870 EVO OEM, MZN* = mSATA/M.2 SATA). Префиксы опасно пересекаются с NVMe-линейкой Samsung.
- `NVME` в модели → NVMe;
- префиксы `HFM` (SK hynix), `KBG` (KIOXIA), `PM9` (Samsung OEM), `MZV`/`MZ1`/`MZQ` (Samsung M.2/U.2 NVMe) → NVMe.

**Шаг 3. Generic USB → SAT IDENTIFY** (если `interface == USB` и имя обобщённое: «Mass Storage Device», «USB Device» и т.п.). Дешёвые мосты отдают имя моста, а не диска. Открывается **второй handle с write-доступом** (SCSI/ATA pass-through требует write), и реальная модель/серийник/прошивка тянутся через `identify_device_via_sat` (см. §4.4).

**Шаг 4. Проверка SMART → уточнение интерфейса.** Для SATA/ATA/USB шлётся `SMART_GET_VERSION`. Для UNKNOWN: **если SMART отвечает → `interface = SATA`**. *Почему:* Intel RST/VMD дают bus_type=Unknown для обычных SATA SSD; NVMe-эвристика уже отработала на шаге 2, значит раз SMART отвечает — это SATA.

## 4. Чтение SMART: четыре транспорта

Файлы: `core/smart_ata.py`, `core/smart_usb_nvme.py`. Здоровье диска читается через четыре разных транспорта в зависимости от типа подключения.

### 4.1. ATA/SATA legacy IOCTL — `read_smart_attributes`

Для внутренних SATA/ATA-дисков, видимых через родной драйвер Windows.

**Команда** строится в структуре `SENDCMDINPARAMS` (с `_pack_ = 1` — строгий бинарный формат): ATA-таскфайл с `Features = sub-команда SMART`, сигнатура SMART `LBA Mid=0x4F / LBA High=0xC2` (без неё диск не распознает SMART-команду), `Command = 0xB0`. **`bDriveNumber = 0` всегда** — на SATA устройство выбирается по handle (`\\.\PhysicalDriveN`), а поле — легаси от IDE master/slave.

Последовательность: `SMART_ENABLE_OPERATIONS` (best-effort, ошибку глотаем) → `SMART_READ_ATTRIBUTES` (0xD0) → `SMART_READ_THRESHOLDS` (0xD1, необязательно).

**Парсинг 512 байт:** данные начинаются с offset 2 (2 байта версии), далее до 30 записей по 12 байт:

```
байт [0]      Attribute ID  (0 = пустой слот, пропускаем)
байт [1-2]    Flags         (uint16 LE)
байт [3]      Current value (нормализованное, 1..253)
байт [4]      Worst value
байт [5-10]   Raw value     (6 байт LE → 48-битное число)
```

### 4.2. Определение HealthLevel по каждому атрибуту

```python
if threshold > 0 and current <= threshold:
    health = CRITICAL                       # достигнут заводской порог отказа
elif threshold > 0 and current < 100 and current <= threshold + 10:
    health = WARNING                        # в 10 пунктах от порога
elif is_critical and (raw & 0xFFFFFFFF) > 0 and attr_id in (5, 196, 197, 198):
    health = WARNING                        # «событийные» атрибуты с ненулевым raw
else:
    health = GOOD
```

- **CRITICAL:** нормализованное `current` упало до/ниже заводского `threshold` — диск официально считает атрибут отказавшим.
- **WARNING (порог рядом):** в пределах 10 пунктов от порога и `current < 100` (значение 100 = норма, не тревожим).
- **WARNING (low32-маска):** для секторных атрибутов **5/196/197/198** значим сам факт ненулевого счётчика. Берутся **только младшие 32 бита** raw — у контроллеров SandForce старшие байты содержат служебный мусор, и без маски здоровый диск получал бы ложный WARNING.

### 4.3. USB-SATA через SAT — `read_smart_via_sat`

USB-карман не отвечает на legacy SMART IOCTL — мост понимает только SCSI/SAT. ATA-команды «заворачиваются» двумя способами, функция автоматически выбирает рабочий:
1. `IOCTL_ATA_PASS_THROUGH` (структура `ATA_PASS_THROUGH_EX`, офсеты через `.offset` — `DataBufferOffset` это `ULONG_PTR`, 8/4 байта);
2. `IOCTL_SCSI_PASS_THROUGH` с **CDB ATA PASS-THROUGH (16), opcode 0x85**, PIO Data-In — более универсальный, работает даже на мостах без поддержки ATA PT.

**Probe:** сначала пробуем `SMART_ENABLE`; если упал — пробуем сразу читать атрибуты (некоторые мосты не пропускают ENABLE, но читают). Парсинг и оценка здоровья — те же, что у legacy.

**`check_scsi_status` — «детектор лжи».** `DeviceIoControl` может вернуть успех (TRUE), при том что сама SCSI-команда внутри завершилась ошибкой (CHECK CONDITION) — статус лежит в поле `ScsiStatus` (**offset 2** заголовка `SCSI_PASS_THROUGH`), не в коде IOCTL. Без этой проверки мусорный буфер трактуется как валидный SMART. Функция проверяет ScsiStatus после **каждой** SCSI-команды (включая DATA_OUT), расшифровывает по именам SAM-5 и прикладывает sense-данные в hex.

### 4.4. Реальное имя USB-кармана — `identify_device_via_sat`

Цепочка из трёх методов получить ATA IDENTIFY DEVICE:
1. **SCSI INQUIRY VPD page 0x89** (ATA Information) — по стандарту SAT мост обязан вернуть копию IDENTIFY с offset 60 страницы. Самый надёжный для USB.
2. `IOCTL_ATA_PASS_THROUGH` с командой 0xEC (важно: LBA Mid/High = 0, **не** сигнатура SMART!).
3. SCSI SAT CDB 0x85 с командой 0xEC.

Декодирование строк (`_ata_string`): ATA IDENTIFY хранит ASCII «словами big-endian» — в каждом 16-битном слове байты переставлены (`'WD'` = байты `'D','W'`), нужен своп пар + strip пробелов И NUL'ов. Офсеты: serial 20-39, firmware 46-53, model 54-93.

### 4.5. USB-NVMe мосты — `read_usb_nvme_smart`

USB-NVMe мост не поддерживает стандартные NVMe IOCTL. SMART достаётся туннелированием NVMe Admin-команды Get Log Page через **vendor-specific SCSI CDB**, у каждого моста свой:
- **JMicron** (JMS583/581) — CDB `0xA1`, **3-шаговый**: отправка NVMe-команды (DATA_OUT, 512 байт с сигнатурой «NVME») → DMA-IN приём данных (DATA_IN, 512 байт) → completion;
- **ASMedia** (ASM2362/2364) — CDB `0xE6`, 1-шаговый;
- **Realtek** (RTL9210/9211/9220) — CDB `0xE4`, 1-шаговый.

Цепочка перебирает JMicron → ASMedia → Realtek; первый давший валидный ответ побеждает. Ответ проверяется `_looks_valid_health_page`: правдоподобность температуры (200-400 K) или spare/threshold (≤100%) — мост может вернуть 512 байт мусора с ненулевыми байтами, наивная проверка `any()` этого не ловит. `check_scsi_status` вызывается и для DATA_OUT — иначе шаг 1 JMicron «молча удавался» бы, а шаг 2 возвращал мусор.

## 5. Чтение NVMe Health

Файл: `core/smart_nvme.py`, функция `read_nvme_health_auto`. Это диспетчер, перебирающий **5 методов** и до **~22 комбинаций**.

### 5.1. Почему так много методов

NVMe-доступ под Windows фрагментирован: зависит от версии Windows/Storage-драйвера, типа контроллера (родной StorNVMe против OEM Intel RST/VMD), того, висит ли диск напрямую или за RAID/SCSI-адаптером, прав доступа (RO/RW) и даже версии SDK (отсюда разные размеры структуры). Код перебирает декартово произведение всех осей, начиная с самого вероятного на Win11.

```
Method 1: IOCTL_STORAGE_QUERY_PROPERTY на \\.\PhysicalDriveN
          2 access mode (RW,RO) × 2 PropertyId (Dev,Adapter) × 3 размера = 12 комбинаций
Method 2: то же через SCSI-адаптер \\.\ScsiN:           = 6 комбинаций
Method 3: IOCTL_STORAGE_PROTOCOL_COMMAND (прямая NVMe Admin)  = 2 комбинации
Method 4: IOCTL_SCSI_MINIPORT + сигнатура NvmeMini       = 1
Method 5: PowerShell/WMI fallback                        = 1
                                                  Итого до 22 попыток
```

На здоровом NVMe в Win11 обычно срабатывает уже первая-вторая комбинация Method 1. Все ошибки копятся, и если ни один метод не прошёл — поднимается `DiskAccessError` со списком всех попыток (для диагностики).

### 5.2. Ключевой принцип: офсеты через `ctypes.sizeof()`, не хардкод

Историческая проблема (баги 44/52/564): `STORAGE_PROTOCOL_SPECIFIC_DATA` в разных версиях SDK имеет разное число полей, а значит разный `sizeof`. Раньше офсеты были захардкожены, и на машине с другой версией структуры данные разъезжались.

Решение — три варианта структуры (`_ProtoData7`=28 байт, `_ProtoData10`=40, `_ProtoData11`=44) + перебор. Буфер `[заголовок query 8][proto-структура][512 байт под health]` собирается через `from_buffer` (наложение структуры поверх буфера без копирования), а `ProtocolDataOffset = ctypes.sizeof(proto)` — снова из sizeof. Позиция данных в ответе тоже вычисляется из того, что **сообщил драйвер** (`ProtocolDataOffset`/`Length` из дескриптора), а не из константы.

**Валидация ответа:** длина данных не 0 и не > 576; начало+длина не вылезают за буфер; первые 32 байта не все нули. Только пройдя все три — парсим.

### 5.3. Парсинг 512-байтного Health Log — `_parse_raw_health`

Раскладка строго по NVMe-спеке (офсеты задаются один раз порядком инкрементального чтения):

| Поле | Offset | Тип | Обработка |
|---|---|---|---|
| critical_warning | 0 | 1 байт | битовая маска (как есть) |
| temperature | 1 | uint16 LE, Кельвины | sanity 200-400 K → K−273, иначе 0 |
| available_spare / threshold / percentage_used | 3/4/5 | 1 байт, % | как есть |
| data_units_read/written, host commands, controller_busy, power_cycles, power_on_hours, unsafe_shutdowns | 32..159 | 128-bit LE | каждое срезом 16 байт |
| media_errors / error_log_entries | 160/176 | 128-bit LE | **& 0xFFFFFFFFFFFFFFFF** |
| warning/critical_temp_time | 192/196 | uint32 LE, минуты | как есть |
| temperature_sensors[0..7] | 200 | 8×uint16, Кельвины | range-check 200-400 K |

**Защиты:**
- **Температура Кельвин→Цельсий с sanity:** прямой минус 273 опасен (0 K дало бы −273 °C). Только `200 ≤ K ≤ 400` (−73..+127 °C) → `K−273`; иначе 0 («неизвестно», GUI игнорирует).
- **Маска `& 0xFFFFFFFFFFFFFFFF`** на счётчиках ошибок: прошивка **SK hynix BC711** пишет мусор в старшие байты 128-битных полей. Число ошибок не может превысить 2⁶⁴, обрезаем до младших 64 бит — иначе вылезали бы «триллионы ошибок» там, где их ноль.
- **8 сенсоров с range-check:** незаполненные (0 K) и битые (0xFFFF) **не попадают** в список (иначе таблица показала бы −273 °C для каждого отсутствующего).

### 5.4. WMI fallback

Когда все IOCTL провалились (типично — USB-NVMe мост без passthrough), `_read_nvme_health_wmi` запускает PowerShell `Get-StorageReliabilityCounter`. `drive_number` жёстко приводится к `int` (подставляется в текст скрипта). Результат проверяется на `isinstance(dict)` (ConvertTo-Json может вернуть массив/скаляр), каждое поле — через `_wmi_int` (число/строка `"50.5"`/null/мусор → int или 0, не падает). Даёт лишь подмножество полей (температура, износ, наработка, циклы, ошибки чтения) и ставит флаг **`wmi_fallback=True`** — чтобы анализ здоровья знал: данные неполные, не делать выводов из нулей.

## 6. Анализ здоровья: Health Score, TBW, WAF

Файл: `core/health_assessor.py`.

### 6.1. Два независимых результата

Оценка сводит **две метрики**:
1. **Health Score (0–100)** — стартует со 100, вычитает штрафы за проблемы.
2. **Уровень (`HealthLevel`)** — GOOD/WARNING/CRITICAL/UNKNOWN — вычисляется **отдельно**, из списков `warnings`/`critical_issues`.

Уровень **не выводится** из score: score может быть 95, но один `critical_issue` даст CRITICAL. Score отражает «общий износ», уровень — «бить ли тревогу сейчас». Соглашение: **`-1` во всех числовых полях = «неизвестно»** (не «ноль») — GUI отличает «0 TB записано» от «не смогли прочитать».

### 6.2. Формула Health Score (ATA)

| Атрибут | ID | Формула штрафа | Cap |
|---|---|---|---|
| Reallocated Sectors | 5 | `min(40, realloc·2)` | 40 |
| Uncorrectable Errors | 187,198 | `min(40, max(187,198)·5)` | 40 |
| Program Fail | 171 | `min(30, ·3)` | 30 |
| Erase Fail | 172 | `min(30, ·3)` | 30 |
| Pending Sectors | 197 | `min(20, ·4)` | 20 |
| SSD Life Left | 231 | ступенчато (см. ниже) | 30 |
| Wear Leveling | 177 | ступенчато (если нет 231) | 25 |
| Temperature | 194/190 | ступенчато (low8) | 15 |
| CRC Errors | 199 | `min(10, crc)` | 10 |

Секторные атрибуты (5/187/196/197/198) берутся через `decoded()` — vendor-профиль или low32-fallback (SandForce пакует счётчики в старшие 32 бита; без маски один сектор превратился бы в астрономическое число). Uncorrectable берёт **максимум** из 187 и 198 (это разные представления одной проблемы, сумма дала бы двойной штраф). Score не уходит в минус (`max(0, score)`).

### 6.3. Защиты от ложных срабатываний — смысловое ядро

Программа не должна врать. Ключевые защиты:

| Защита | Проблема, которую решает |
|---|---|
| Пустые атрибуты → UNKNOWN, score −1 | не показывать фальшивые 100/100, когда SMART не прочитан |
| **SSD Life Left (231): штраф только если `life_left>0` ИЛИ `raw>0`** | Kingston/Silicon Motion (KC600, SM2259) репортят 231=всё-нули (заглушка) → давало ложный «Износ SSD: 100%» −25 баллов новому диску |
| Wear Leveling (177) только если нет ID 231 | не штрафовать износ дважды |
| Температура `& 0xFF` (low8) | min/max в старших байтах raw → ложные сотни °C |
| low32 для секторных без профиля | SandForce пакует счётчики в старшие 32 бита |
| WARNING-близость к порогу только при `current<100` | 100 = норма, не тревожить |
| **NVMe OEM-драйвер → UNKNOWN** | Intel RST/VMD: WMI отдаёт только температуру (POH=0, DUW=0, циклы=0) → иначе ложный GOOD 100/100 |
| Available Spare только при `!wmi_fallback` | WMI не отдаёт корректный spare/threshold |
| WAF NVMe = −1 | стандарт не даёт NAND-writes, нечем честно посчитать |
| Cap remaining_days = 40000 (~110 лет) | почти нулевая запись → миллионы дней мусора |
| ID 202 override (Crucial) → critical=False | «Address Mark Errors» (HDD) vs «остаток ресурса» (SSD) |

### 6.4. TBW и WAF

**Consumed TBW:** ID 241 (Total LBAs Written × 512) или fallback ID 233 (32 MB units). **Rated TBW:** эвристика `capacity_tb × 600` (600 TBW/TB для consumer TLC) — это **не** спека производителя, поле помечается `tbw_estimation_method="heuristic_600_per_tb"` (QLC ~150, enterprise ~3000+).

**Прогноз:** `daily_write = consumed / (POH/24)`, `remaining_days = min(remaining_tb / daily_write, 40000)`. Строится только при наработке > 24 ч.

**WAF (Write Amplification Factor)** = NAND-записи / Host-записи (идеал ~1.0, сборка мусора/SLC-кэш повышают): ID 249 (NAND GiB) vs 241, или fallback ID 243 vs 241. Для NVMe WAF всегда −1 (стандарт не даёт NAND-writes). HDD: TBW не считается (ранний return).

### 6.5. NVMe Health Score

Штрафы: Critical Warning (30, фикс), Media Errors (`min(40, ·5)`), Percentage Used (ступенчато до 30), Available Spare (20/10, только не-WMI), Temperature (до 15, пороги выше ATA — у NVMe нормальные рабочие температуры выше), Unsafe Shutdowns (10/5), Critical Temp Time (10). Critical Warning разбирается побитово в человекочитаемые причины (spare ниже порога, перегрев, надёжность снижена, read-only, backup failure).

## 7. Vendor-профили: декодирование и переопределение имён

Файлы: `data/vendor_profiles.py`, `data/smart_db.py`.

### 7.1. Декодирование packed raw

Некоторые контроллеры **упаковывают** несколько значений в одно 64-битное raw-поле. Профиль описывает, как извлечь полезную часть:

| Метод | Маска | Назначение |
|---|---|---|
| `raw` | как есть | стандартный контроллер |
| `low8` | `& 0xFF` | температура (min/max в старших байтах) |
| `low16` | `& 0xFFFF` | младшие 2 байта |
| `low20` | `& 0xFFFFF` | SandForce Power-On Hours |
| `low32` | `& 0xFFFFFFFF` | SandForce критические счётчики |

`match_profile(model, firmware)` ищет профиль по подстроке модели/прошивки (первое совпадение, порядок важен). `decode_raw` применяет правило профиля, либо `_DEFAULT_DECODE` (**температура 190/194 всегда low8, даже без профиля** — она пакуется почти у всех), либо возвращает raw как есть.

8 профилей: SandForce SF-2281 (самый «тяжёлый» — почти все атрибуты packed), Kingston A400/NV2, Transcend, Intel, Samsung, SanDisk, Crucial/Micron MX.

### 7.2. Переопределение имён (per-vendor) — `get_attribute_override`

Один SMART ID у разных вендоров значит разное. Канонический пример — **ID 202**:
- стандартно = «Data Address Mark Errors» (ошибка, критичный атрибут HDD);
- у **Crucial/Micron** = «Percentage Lifetime Remaining» (остаток ресурса SSD: 100=новый, 0=исчерпан — **это не отказ**).

Без override инструмент паниковал бы на здоровом Crucial. Профиль несёт поле `"names": {202: {name_en, name_ru, desc_en, desc_ru, critical: False}}`, и функции `get_attribute_name/description/is_critical(attr_id, override)` отдают ему приоритет над базой. База `smart_db.py` при этом **не зависит** от vendor_profiles — принимает готовый dict (слабая связность).

### 7.3. База атрибутов `smart_db.py`

`SmartAttributeInfo` (frozen dataclass): `name_en/ru`, `desc_en/ru`, `is_critical`, `unit`. Свойства `.name`/`.description` выбирают язык через `tr()`. ~80 атрибутов в словаре `SMART_ATTRIBUTES`. Множество `SSD_INDICATOR_ATTRS` (23 ID) — признаки SSD.

## 8. Движок бенчмарка

Файлы: `core/benchmark.py`, `data/baselines.py`. Измерение производительности через **raw I/O мимо файловой системы** (`\\.\PhysicalDriveN` + `FILE_FLAG_NO_BUFFERING`).

### 8.1. Профили

| Профиль | Write | SLC | Фазы записи |
|---|---|---|---|
| **quick** | нет | нет | только чтение |
| **standard** | да | нет | seq/random/mixed/verify (без SLC) |
| **full** | да | 50 GB | все 5 |
| **stress** | да | 100 GB | все 5, verify 1 GiB |

Флаг `include_slc` отделён от `include_write`, чтобы Standard мог делать запись, но не запускать долгий SLC-тест.

### 8.2. Фазы

**Чтение (неразрушающее):**
- **Sequential Read** — 1 warm-up прогон (отбрасывается) + 3 измеряемых → **медиана** (устойчивее среднего к выбросам ОС).
- **Random 4K Read (QD1)** — 5000 случайных чтений, latency каждого в мкс, перцентили P95/P99/P99.9/P99.99. Флаг `random_low_sample = n<10000` сигнализирует, что хвостовые перцентили статистически слабы (при 5000 P99.99 опирается на 1 выборку).
- **Full Drive Read Sweep** — 200 точек по диску, график «скорость vs позиция» (для HDD виден спад от внешних к внутренним дорожкам).

**Запись (деструктивное):** Sequential Write, Random 4K Write, Mixed I/O 70/30 (30 сек), Write-Read-Verify (запись→чтение→сравнение MD5), SLC Cache Test (детекция «обрыва» кэша).

### 8.3. Методология QD1 — почему это latency-тест

Все случайные тесты идут строго синхронно: одна операция «в полёте», цикл ждёт её завершения. Это измеряет **время отклика** (latency), а IOPS = `1/средняя_латентность`. **Нельзя сравнивать с QD32-спеками производителей:** маркетинговые IOPS заявляются при глубине очереди 32+ (десятки операций параллельно, контроллер скрывает латентность по множеству каналов NAND). Синхронный QD1 физически не достигает таких чисел. Поэтому baseline хранит QD1-диапазоны, а маркетинговые QD32-пики — отдельно (`qd32_peak_*`), только как справку. QD1-latency — это то, что реально чувствует пользователь (отзывчивость системы).

### 8.4. Безопасность — три рубежа

**1. Защита зоны MBR/GPT/EFI (1 GiB).** `MBR_PROTECT_BYTES = 1 GiB`. **ВСЕ** write-фазы стартуют не раньше этого смещения: последовательные (seq_write, verify, slc) — `seek(1 GiB)`; случайные (rnd_write, mixed) — `randrange(1 GiB, max)`. Самый тонкий случай — **verify** открывает диск дважды (запись и чтение — разные хэндлы), поэтому `seek(1 GiB)` нужен в **обеих** фазах, иначе чтение пошло бы с LBA 0 и сравнение дало бы 100% mismatch. На малых дисках (< ~1.5 GB) фазы **пропускаются** с предупреждением, а не пишут в защищённую зону.

**2. Volume lock fail-closed.** Перед записью `lock_and_dismount_volumes`; если `failed_volumes` непустой → **abort** (не пишем вообще). Unlock гарантирован в `finally`.

**3. Direct I/O без кэшей.** `FILE_FLAG_NO_BUFFERING` (обход кэша Windows) + `FILE_FLAG_WRITE_THROUGH` (обход write-back кэша контроллера) + `AlignedBuffer`. **Антикомпрессия:** на каждой write-итерации новый `os.urandom()` — иначе контроллеры с компрессией/дедупликацией (SandForce) распознают повторяющийся паттерн и нарисуют завышенную скорость, не отражающую реальную нагрузку на NAND.

**Temperature polling** во всех фазах (не чаще раза в 5 сек) — позволяет отличить thermal throttling от реального обрыва SLC-кэша.

### 8.5. SLC Cache Test

Льёт непрерывную запись порциями по 100 MiB, замеряя скорость каждой. **Cliff** детектируется в реальном времени: когда среднее последних 3 точек падает ниже `initial × SLC_CLIFF_RATIO (0.6)` (60% от начальной) — кэш исчерпан, дописываем ещё 3 GB для измерения устойчивой скорости прямого NAND и останавливаемся. `slc_cache_size_gb` = объём до обрыва.

### 8.6. Сравнение с эталонами

`detect_class` определяет класс по интерфейсу + seq_read (порог 250 MB/s отделяет SATA SSD от HDD, 4000 — Gen4 от Gen3). `compare_to_baseline` даёт вердикт pass/warn/fail (`pass` если ≥ нижней границы, `warn` если ≥ 60% от неё). IOPS сравниваются **только** с QD1-диапазонами + примечание «QD1 latency-тест — спеки производителя обычно для QD32» (защита от ложной паники).

## 9. Сканирование поверхности (Surface Scan)

Файл: `core/surface_scan.py`. Аналог Victoria HDD: последовательное чтение всей поверхности с замером latency каждого блока и классификацией по «светофору».

### 9.1. Режимы и категории

**Режимы (`ScanMode`):**
- **Ignore** — только чтение, классификация; при ошибке drill-down (идентификация битых LBA), но **ничего не пишет**. Безопасный режим по умолчанию.
- **Erase** — нули **только в битые секторы** (drill-down); опционально (`+slow`) затирает и медленные блоки. Хорошие данные целы.
- **Refresh** — чтение → перезапись **тех же** данных (освежает деградирующие секторы HDD без потери данных).
- **Write** — нули на ВСЮ поверхность без чтения (полное стирание).

**Категории по latency:** EXCELLENT <5ms, GOOD <20, ACCEPTABLE <50, SLOW <150, VERY_SLOW <500, CRITICAL ≥500, ERROR (исключение). Имитация шкалы Victoria: растущая latency на HDD = рекалибровка головок и повторные чтения слабого сектора.

### 9.2. Валидация диапазона (fail-fast)

В конструкторе, **до открытия диска**: `ValueError` при `start<0`, `end<0`, `capacity≤0`, `block_size≤0`, `start≥end`, `end>capacity`. Offset выравнивается по `block_size` (требование `NO_BUFFERING`); `end_offset=0` трактуется как «до конца диска».

### 9.3. Основной цикл

**Volume lock fail-closed** для пишущих режимов (abort при failed_volumes), **try/finally гарантирует unlock** (основной цикл вынесен в `_do_scan`, чтобы lock-логика была компактна).

**Ключевой приём — последовательное чтение БЕЗ seek.** `ReadFile` сам продвигает файловый указатель ОС, поэтому между блоками seek не нужен. `seek` вызывается **только** после ошибки или записи (когда указатель сбился), по флагу `need_seek`. *Почему:* лишний `SetFilePointerEx` перед каждым блоком ломает аппаратный read-ahead контроллера и снижает последовательную скорость.

**Аварийный стоп — 100 ошибок подряд.** `consecutive_errors` сбрасывается при любом успешном блоке, поэтому диск с разбросанными единичными бэдами сканируется до конца; стоп — только при сплошной полосе из 100 ошибок (тотальный отказ; каждая ошибка стоит секунды time-out контроллера).

### 9.4. Drill-down — посекторный анализ

Когда блок (1 MB) не прочитался, неизвестно, какой сектор битый. Drill-down перечитывает блок по 4096 байт (безопасно для 4Kn-дисков), находит конкретные нечитаемые секторы (LBA в традиционных 512-байтных единицах), и в режимах Erase/Refresh/Write пишет нули **только в битые секторы** — данные пользователя сохраняются. `bad_sector_callback` подсвечивает битый LBA в GUI сразу. *Почему запись «лечит»:* на HDD запись в сбойный сектор заставляет контроллер перемагнитить его «начисто» или выполнить **remap** на резервный сектор.

### 9.5. Почему для SSD это HDD-centric

На SSD выводы surface scan нужно интерпретировать осторожно:
- **LBA ≠ физическая ячейка** — между ними стоит FTL контроллера; «битый LBA» не указывает на конкретную сбойную ячейку.
- **latency не отражает износ** — определяется очередями контроллера, фоновым GC, SLC-кэшем, а не здоровьем ячейки.
- **запись «для лечения» вредна** — расходует ресурс P/E-циклов, а bad-block management SSD делает сам контроллер прозрачно.

Для SSD методологически правильный инструмент — **SMART/NVMe health** (Percentage Used, Media Errors), а не surface scan. Режим Ignore полезен как грубая проверка читаемости всех LBA.

## 10. GUI и многопоточность

Файлы: `disk_diag/app.py`, `gui/main_window.py`, `gui/smart_table.py`.

### 10.1. Запуск

`run.py` проверяет права админа **до импорта PySide6** (ленивый импорт): `is_admin()` → если нет, через `QMessageBox` спрашивает и перезапускается через `ShellExecuteW` с операцией `runas` (триггер UAC), передавая исходные `sys.argv` (флаги CLI сохраняются). `app.py`: `QApplication`, стиль Fusion (одинаков на всех Windows, полностью кастомизируется через QSS), тёмная тема Catppuccin Mocha, и **`showMaximized()`** (см. §10.4).

### 10.2. Qt threading и защита от гонок

SMART-чтение медленное (особенно USB) — выполняется в фоне через паттерн «worker-object + QThread» (`_SmartWorker(QObject)` + `moveToThread`). Старый поток перед новым запуском: `quit()` + `wait()` + **`deleteLater()`** (без явной утилизации Qt-объекты копятся до закрытия окна).

**Generation-счётчик — защита от «тихого ложного диагноза».** Пользователь быстро переключает диски; сигнал `finished` от **старого** воркера может прийти после выбора нового диска, и SMART диска А отрисовался бы под именем диска Б. Решение: `_smart_gen` инкрементируется при каждом запуске, поколение фиксируется в лямбде (`g=gen` в дефолтном аргументе), и обработчик отбрасывает результат, если `gen != self._smart_gen`.

**closeEvent** при активном benchmark/surface-scan показывает подтверждение «Прервать тест и выйти?» — тихо бросить поток с незавершённой raw-записью нельзя (повреждение данных). `stop()` шлёт `cancel()` и ждёт до 3+15 секунд.

### 10.3. Логика чтения SMART в воркере

Ветвление по интерфейсу: **NVMe** → `read_nvme_health_auto`; **USB** → трёхступенчатая цепочка (USB-SATA SAT → USB-NVMe мосты → стандартный NVMe IOCTL → none); **ATA** → `read_smart_attributes`. **Виртуальные диски** перехватываются ещё в главном потоке (поток не запускается) — показывается подсказка использовать инструменты гипервизора (smartctl/storcli/zpool/mdadm).

### 10.4. Ширина колонок — три слоя против «залипания» Stretch

Колонка «Атрибут» (ATA) / «Параметр» (NVMe) = **Stretch** (растягивает таблицу на всю ширину окна). Но Stretch вычисляет ширину от текущей ширины таблицы в момент заполнения. При **асинхронном** заполнении из worker-потока ширина ещё «переходная», и Stretch застывает на узком значении (таблица слева, чёрная пустота справа, имена обрезаны). Решение — три независимых слоя применения:
1. **синхронно** сразу после заполнения;
2. **отложенно** `QTimer.singleShot(0)` — выполнится после применения финальной геометрии окна;
3. **`resizeEvent`** — переприменяет при любом ресайзе окна.

Плюс ранний `showMaximized()` — окно сразу широкое, таблица верстается на полной ширине с первого кадра.

Описания атрибутов хранятся в `UserRole` ячейки (а не по индексу строки) — при сортировке «едут» вместе с атрибутом. Критические подсвечены синим `#89B4FA`.

## 11. CLI и модель безопасности

Файл: `cli.py`. Команды: `--list`, `--smart N` (+`--json`), `--benchmark N`, `--history SERIAL`. Проверка `args.smart is not None` (диск 0 валиден, `if args.smart:` отбросил бы ноль).

**Многоуровневая защита `--benchmark --write`:**
1. **Авто-апгрейд** quick→standard (quick read-only по смыслу).
2. **Печать плана записи** с явным «All existing data on PhysicalDrive{N} will be DESTROYED».
3. **Защита системного диска:** `is_system_drive` → отказ (exit 2), если нет `--force-system-drive`.
4. **Подтверждение по серийнику:** пользователь вводит точный серийник диска (или `DESTROY`, если серийника нет). Несовпадение → exit 3.
5. **TOCTOU-защита (re-enumerate):** между подтверждением и стартом пользователь мог переткнуть USB — номера дисков сдвигаются, и запись ушла бы на ДРУГОЙ диск. Список перечитывается, серийник под тем же номером сверяется; диск исчез/сменился → exit 4.

Коды возврата: 0 успех, 1 диск не найден/нет SMART, 2 системный диск без флага, 3 несовпадение подтверждения, 4 TOCTOU, 130 Ctrl+C.

### 12.1. Локализация (`i18n.py`)

`tr(en, ru)` — обе строки инлайн в коде, возвращается нужная по `_lang` (нет внешних .po/.json — минимализм). `lang.cfg` рядом с exe (через `sys.executable` при PyInstaller) или в корне проекта при разработке (`getattr(sys, 'frozen', False)`). Смена языка требует перезапуска (`tr()` вычисляется при построении виджетов).

### 12.2. История тестов (`history.py`)

SQLite `disk_history.db` (рядом с exe/в корне проекта). Все операции через `contextlib.closing` — гарантирует `db.close()` даже при ошибке SQL (иначе соединения копились бы в долгоживущем GUI). `save_test` сохраняет penalties как JSON (`ensure_ascii=False` — кириллица читаема). Запись истории в `try/except` — ошибка не роняет диагностику.

### 12.4. Экспорт

Три формата через `QFileDialog`: SMART txt (Ctrl+S, со склонением «балл/балла/баллов»), Benchmark txt (Ctrl+B, с температурным логом), JSON (Ctrl+J, полная сессия, `ensure_ascii=False`). Все в `try/except OSError`.

## 13. Сквозные инженерные принципы

Эти принципы повторяются во всех подсистемах и формируют «характер» программы:

1. **Ноль внешних зависимостей для дисков** — только ctypes + kernel32. WMI/PowerShell лишь как последний fallback.
2. **`_pack_ = 1` только для ATA/SMART-структур** (фиксированный бинарный формат «провода»); Storage API структуры — нативное выравнивание Windows. Перепутать = битые офсеты.
3. **Дисциплина `sizeof`** — никогда не хардкодить размеры/офсеты Windows-структур (они плавают между версиями SDK; исторические баги 44/52/564).
4. **Fail-safe везде** — `is_admin`, история, экспорт, локализация в `try/except`; ошибка инфраструктуры не роняет диагностику. Один проблемный диск не валит перечисление.
5. **Fail-closed для деструктива** — volume lock: если хоть один том не заблокирован, запись отменяется. MBR/GPT-зона (1 GiB) защищена во всех write-фазах.
6. **Дисциплина закрытия ресурсов** — `contextlib.closing` (SQLite), `deleteLater` (Qt), `DeviceHandle`/`AlignedBuffer` как context manager. Не копить открытые хэндлы.
7. **Защита от ложных срабатываний** — масса проверок против ложного GOOD (мёртвый диск, пустой WMI), ложного износа (Kingston 231-заглушка), ложного перегрева (packed-температура), ложных ошибок (SK hynix мусор в старших байтах). Программа предпочитает честный UNKNOWN ложному выводу.
8. **Защита от гонок и TOCTOU** — generation-счётчик в GUI (stale SMART), re-enumerate в CLI (сдвиг номеров USB): «состояние изменилось между намерением и действием».
9. **Множество fallback-цепочек** — NVMe (~22 попытки), USB (SAT → мосты → IOCTL → WMI), ёмкость (3 метода), IDENTIFY (VPD → ATA PT → SAT). Переживает отказ любого отдельного метода.
10. **Честная методология** — QD1-latency строго отделён от маркетинговых QD32-спеков; TBW помечен как эвристика, а не спека; surface scan для SSD честно назван HDD-centric.
